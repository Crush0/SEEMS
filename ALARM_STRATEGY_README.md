# 报警策略配置系统 - 使用指南

## 系统概述

报警策略配置系统为SEEMS提供了灵活的报警管理能力，管理员可以通过界面动态配置各种报警策略，包括阈值设置、触发条件、触发时机、消息通知及接收人配置。

## 功能特性

### 1. 多种触发条件
- **低于 (LESS_THAN)**: 当值低于设定阈值时触发
- **高于 (GREATER_THAN)**: 当值高于设定阈值时触发
- **区间 (BETWEEN)**: 当值在设定区间内时触发
- **等于 (EQUAL)**: 当值等于设定阈值时触发

### 2. 灵活的触发时机
- **立即触发 (IMMEDIATE)**: 条件满足时立即创建报警
- **持续触发 (DURATION)**: 条件持续满足指定时间后才创建报警（避免瞬时波动误报）

### 3. 多样化的通知方式
- 支持按角色组发送（管理员、操作员、普通用户）
- 支持按具体用户发送
- 自动发送系统消息
- 实时推送（Socket.IO）

### 4. 策略优先级
- 船舶特定策略优先于全局策略
- 可设置优先级数值（0-100，数值越大优先级越高）

## 快速开始

### 1. 启用报警策略系统

编辑 `seems-script/config.yaml`:

```yaml
alarm:
  # 启用策略配置
  use_strategy: true

  # 默认阈值（未配置策略时使用）
  soc_threshold: 15.0
  temp_threshold: 60.0
```

### 2. 访问策略管理页面

登录系统后，导航到：**报警管理** → **报警策略**

### 3. 创建报警策略

点击"新增策略"按钮，填写以下信息：

#### 基本信息
- **船舶**: 选择特定船舶或留空创建全局策略
- **报警类型**: 选择低电量、高温、GPS丢失等类型
- **报警级别**: 选择信息、警告、错误、严重
- **优先级**: 设置策略优先级（0-100）

#### 触发条件
- **触发条件**: 选择低于、高于、区间、等于
- **阈值1**: 输入第一个阈值
- **阈值2**: 区间条件时需要输入第二个阈值

#### 触发时机
- **触发时机**: 选择立即触发或持续触发
- **持续时间**: 持续触发时需要设置持续时间（秒）

#### 消息通知
- **发送消息**: 是否发送消息通知
- **标题模板**: 报警标题模板（支持变量）
- **内容模板**: 报警内容模板（支持变量）

#### 接收人配置
- 点击"添加接收人"
- 选择接收人类型（角色组/具体用户）
- 选择具体的角色或用户

### 4. 模板变量

在标题和内容模板中可以使用以下变量：

| 变量名 | 说明 | 示例值 |
|--------|------|--------|
| `{ship_name}` | 船舶名称 | 拖轮1 |
| `{alarm_type}` | 报警类型 | 低电量报警 |
| `{value}` | 当前值 | 12.5 |
| `{threshold}` | 阈值 | 15.0 |
| `{alarm_time}` | 报警时间 | 2026-03-10 15:30:00 |
| `{battery_position}` | 电池位置 | 左电池 |
| `{suggestion}` | 建议措施 | 立即充电 |

#### 示例模板

**标题模板**:
```
{ship_name} {alarm_type} - {battery_position}
```

**内容模板**:
```
船舶 {ship_name} 的 {battery_position} 电量过低
当前值: {value}%
阈值: {threshold}%
报警时间: {alarm_time}
建议措施: {suggestion}
```

## 使用场景

### 场景1: 低电量报警

创建一个低电量报警策略：
- **船舶**: 全局策略
- **报警类型**: 低电量报警
- **报警级别**: 严重
- **触发条件**: 低于 15%
- **触发时机**: 立即触发
- **接收人**: 管理员

### 场景2: 高温报警（持续触发）

创建一个高温报警策略：
- **船舶**: 拖轮1
- **报警类型**: 高温报警
- **报警级别**: 错误
- **触发条件**: 高于 60°C
- **触发时机**: 持续30秒后触发（避免瞬时波动）
- **接收人**: 操作员

### 场景3: 区间报警

创建一个电量区间报警：
- **触发条件**: 区间 20% - 25%
- **用途**: 提前预警，电量进入低电量区间

## 技术实现

### 后端API

所有策略管理API都位于 `/api/alarm-strategy` 路径下：

- `POST /create` - 创建策略
- `POST /update` - 更新策略
- `POST /delete` - 删除策略
- `POST /list` - 查询策略列表
- `GET /detail` - 获取策略详情
- `POST /enable` - 启用/禁用策略
- `POST /apply` - 应用策略（Python调用）

### Python集成

Python脚本（`realtime.py`）会自动：
1. 从后端获取有效的报警策略
2. 检查当前值是否满足触发条件
3. 根据触发时机决定是立即报警还是追踪持续时间
4. 创建报警并发送通知

### 性能优化

- **策略缓存**: 策略配置在Redis中缓存5分钟
- **异步处理**: 消息发送异步执行，不阻塞主流程
- **防重复机制**: 10分钟内相同类型的报警只触发一次

## 常见问题

### Q: 如何创建全局策略？

A: 在"船舶"字段留空即可，全局策略适用于所有船舶。

### Q: 船舶特定策略和全局策略的优先级？

A: 船舶特定策略优先于全局策略。如果同一船舶同时存在特定策略和全局策略，系统会使用特定策略。

### Q: 如何避免频繁报警？

A: 有两种方式：
1. 使用"持续触发"时机，设置合理的持续时间
2. 系统内置防重复机制，10分钟内相同报警只触发一次

### Q: 模板变量不生效怎么办？

A: 确保：
- 变量名拼写正确
- 使用花括号包裹变量：`{variable_name}`
- 变量在当前报警场景中可用

### Q: 如何禁用某个策略？

A: 在策略列表中点击"禁用"按钮，或编辑策略后取消勾选"启用"选项。

## 注意事项

1. **向后兼容**: 未配置策略时，系统会使用`config.yaml`中的默认阈值
2. **权限控制**: 只有管理员角色可以管理报警策略
3. **数据验证**: 系统会自动验证阈值的合理性（如BETWEEN条件要求阈值1 < 阈值2）
4. **消息推送**: 确保Socket.IO服务正常运行以接收实时通知
